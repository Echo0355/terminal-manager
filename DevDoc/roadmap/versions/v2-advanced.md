# v2.0 高级能力

## 1. 版本目标

v2.0 的目标是探索高级终端能力和差异化能力。

这些能力复杂度高、风险高，不应在 MVP、v1.0 或 v1.5 阶段提前实现。

v2.0 不是一个必须一次完成的大版本，应拆成多个独立模块逐个评估。

## 2. 版本原则

- 每个高级能力都必须先做技术验证。
- 不破坏已有本地终端管理能力。
- 涉及自动执行、远程连接、插件执行的功能必须有明确安全模型。
- 真正会话保活不能只依赖 `node-pty`。
- 可选能力默认关闭。

## 3. 功能候选

### 3.1 Shell 集成

目标：

提高 cwd、命令开始、命令结束、退出码等状态识别准确度。

候选方案：

- OSC 7：cwd 识别。
- OSC 133：命令生命周期。
- shell profile 注入片段。
- 用户手动开启集成。

验收：

- 能识别 cwd 变化。
- 能 best-effort 识别命令开始/结束。
- 不支持的 shell 自动降级。
- 用户可关闭 Shell 集成。

风险：

- 不同 shell 行为差异大。
- 用户 profile 被修改会引发信任问题。
- 退出码识别不能过度承诺。

### 3.2 后台任务完成通知

依赖：

- Shell 集成。

功能：

- 当前 pane 失焦或应用后台时，命令结束后发送通知。
- 支持通知开关。
- 支持最小时长阈值，例如运行超过 10 秒才通知。

验收：

- 不会对每条短命令刷通知。
- 通知内容不泄露敏感命令，或可配置。

### 3.3 输入广播

功能：

- 将输入同步发送到多个 pane。
- 明确广播模式状态。
- 支持选择目标 pane。
- 支持危险提示。

验收：

- 用户能清楚知道当前正在广播输入。
- 默认关闭。
- Esc 或明显按钮可退出广播。

风险：

- 误把危险命令发送到多个终端。
- 粘贴大段内容造成不可控执行。

### 3.4 SSH 管理

功能：

- 保存 SSH 连接配置。
- 从 UI 创建 SSH 终端。
- 支持 SSH 分组。
- 可选支持密钥路径。

安全要求：

- 不明文保存密码。
- 优先使用系统 ssh 和系统密钥管理。
- 敏感信息不要写入日志。

验收：

- 能从配置打开 SSH 会话。
- 连接失败有错误提示。

### 3.5 真正会话保活

目标：

应用退出后，终端中的进程仍可继续运行，并可重新连接。

可选方案：

- 集成 `tmux`。
- 集成 `zellij`。
- 自研守护进程。

推荐：

先做 `tmux` / `zellij` 集成，不优先自研守护进程。

验收：

- 启用后，应用关闭不杀死后端会话。
- 重启应用可重新连接会话。
- 未启用时保持 MVP 行为。

风险：

- Windows 原生体验复杂。
- 会话、pane、layout 和后端 multiplexer 的映射复杂。

### 3.6 插件系统

功能：

- 插件 manifest。
- 插件加载和禁用。
- 插件 API。
- 支持主题类插件。
- 支持命令类插件。

安全要求：

- 插件权限声明。
- 插件来源限制。
- 插件错误隔离。

验收：

- 插件崩溃不影响终端核心能力。
- 用户可以禁用插件。

### 3.7 终端录制

功能：

- 录制终端输入输出。
- 保存为 asciinema 或项目自定义格式。
- 回放录制。

验收：

- 录制不明显影响终端性能。
- 可暂停和停止。

### 3.8 命令块 UI

目标：

类似 Warp，把命令和输出组织成块。

要求：

- 强依赖 Shell 集成。
- 先做实验模式。
- 不替代普通终端渲染。

风险：

- 极大改变终端交互模型。
- 与全屏程序、TUI 程序兼容性差。

## 4. 不建议做的内容

v2.0 也不建议一次性做：

- 自研远程协作协议。
- 自研完整 shell。
- 试图完全解析所有终端输出。
- 默认启用插件执行。
- 默认自动注入用户 shell profile。

## 5. 推荐实施顺序

v2.0 应分模块推进：

1. Shell 集成技术验证。
2. Shell 集成可选开关和 cwd 识别。
3. 命令生命周期 best-effort 识别。
4. 后台任务完成通知。
5. 输入广播。
6. SSH 管理。
7. tmux / zellij 会话保活验证。
8. 终端录制。
9. 插件系统。
10. 命令块 UI 实验模式。

插件系统和命令块 UI 放最后。

## 6. 测试要求

- Shell 集成需要按 shell 类型测试。
- 输入广播需要危险操作保护测试。
- SSH 不能泄露敏感信息到日志。
- 会话保活需要进程生命周期测试。
- 插件系统需要错误隔离测试。
- 所有高级能力都要有关闭开关。

## 7. 验收标准

v2.0 不要求所有候选能力都完成。每个模块独立验收。

一个模块可发布需要满足：

- 默认关闭或安全默认。
- 出错不影响普通终端。
- 有用户文档。
- 有已知限制说明。
- 有基本测试。
- 打包后可用。

## 8. 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| Shell 集成过度承诺 | 状态误判 | 文档明确 best-effort，可关闭。 |
| 输入广播误操作 | 多终端执行危险命令 | 明显视觉提示，默认关闭，确认机制。 |
| SSH 凭据泄露 | 高安全风险 | 不保存密码，敏感信息不进日志。 |
| 插件执行不受控 | 供应链风险 | 权限模型、禁用机制、来源限制。 |
| 会话保活架构复杂 | 破坏现有布局模型 | 先 tmux/zellij 验证，再考虑深度集成。 |
