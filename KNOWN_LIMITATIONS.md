# 已知限制

本文档记录了当前 MVP 版本的已知限制和边界情况。

## 不支持会话保活

**限制：** 关闭应用后，所有正在运行的终端进程会被终止。重启后会按保存的 Shell 和工作目录重新创建终端，但不会恢复之前正在运行的命令或 Shell 内存状态。

**原因：** `node-pty` 只能管理 PTY 进程本身，无法保存和恢复 Shell 的内部状态。

**替代方案：** 如果需要真正的会话保活，可以使用：
- `tmux`：终端多路复用器
- `zellij`：现代终端多路复用器
- 远程守护进程方案

**示例：**
```bash
# 在终端中使用 tmux
tmux new -s my-session

# 关闭应用后重新打开，恢复 tmux 会话
tmux attach -t my-session
```

## 不支持命令自动执行

**限制：** 布局恢复时不会自动执行历史命令。即使保存了某个面板之前运行的命令，重启后也不会自动执行。

**原因：** 自动执行命令存在安全风险，可能导致意外操作。

**未来计划：** 如果支持启动命令，会提供确认机制或安全策略。

## Shell 状态识别不可靠

**限制：** 应用无法可靠地识别 Shell 的内部状态，包括：
- 当前工作目录（只能通过启动参数设置）
- 命令是否正在执行
- 命令的退出码
- 命令的开始和结束时间

**原因：** Shell 状态识别依赖于输出解析或 Shell 集成，这些都是 best-effort 的。

**影响：**
- 状态栏显示的工作目录可能不准确
- 无法可靠地检测命令完成
- 无法获取命令的退出码

## 不支持远程连接

**限制：** 当前版本不支持 SSH、Telnet 等远程连接。

**原因：** 远程连接需要额外的协议实现和安全考虑。

**替代方案：** 可以在终端中直接使用系统自带的 SSH 客户端：
```bash
ssh user@remote-host
```

## 不支持插件系统

**限制：** 当前版本不支持插件扩展。

**原因：** 插件系统需要完整的 API 设计和安全沙箱。

## 不支持命令块 UI

**限制：** 当前版本使用传统的终端输出方式，不支持 Warp 风格的命令块 UI。

**原因：** 命令块 UI 需要 Shell 集成来识别命令边界。

## 不支持输入广播

**限制：** 当前版本不支持同时在多个面板输入相同内容。

**原因：** 输入广播需要额外的 UI 设计和状态管理。

## 不支持浮动面板

**限制：** 当前版本不支持 Zellij 风格的浮动面板。

**原因：** 浮动面板需要额外的布局管理层。

## 不支持面板拖拽重排

**限制：** 当前版本不支持用鼠标拖拽面板在布局树中移动位置。

**原因：** 拖拽重排需要复杂的交互逻辑和状态同步。

## 不支持主题编辑器

**限制：** 当前版本只支持深色和浅色两种内置主题，不支持自定义主题。

**原因：** 主题编辑器需要完整的 UI 和配置系统。

## 不支持全局热键

**限制：** 当前版本不支持全局热键呼出/隐藏窗口（Quake 模式）。

**原因：** 全局热键需要系统级权限和跨平台适配。

## 不支持命令面板

**限制：** 当前版本不支持 Ctrl+Shift+P 命令面板。

**原因：** 命令面板需要完整的命令注册和模糊搜索系统。

## 不支持快速打开

**限制：** 当前版本不支持 Ctrl+P 快速打开项目面板。

**原因：** 快速打开需要完整的项目索引和模糊搜索系统。

## 不支持右键菜单

**限制：** 当前版本不支持终端面板和标签栏的右键菜单。

**原因：** 右键菜单需要额外的 UI 组件和事件处理。

## 不支持终端内容搜索

**限制：** 当前版本不支持在终端输出中搜索文本。

**原因：** 虽然 xterm.js 支持搜索插件，但需要额外的 UI 集成。

## 不支持字体连字

**限制：** 当前版本不支持 Cascadia Code、Fira Code 等编程字体的连字效果。

**原因：** 字体连字需要额外的配置选项和字体检测。

## 不支持自动更新

**限制：** 当前版本不支持应用自动更新。

**原因：** 自动更新需要完整的更新服务器和版本管理系统。

## 不支持 macOS 签名和公证

**限制：** 当前版本的 macOS 构建没有代码签名和公证。

**原因：** 代码签名需要 Apple 开发者账号和证书。

**影响：** macOS 用户首次打开时可能会看到安全警告，需要手动允许运行。

## 不支持 Linux

**限制：** 当前版本只支持 Windows 和 macOS，不支持 Linux。

**原因：** 虽然 Electron 和 node-pty 支持 Linux，但需要额外的测试和适配。

## 性能限制

**限制：** 大量终端输出时可能会出现卡顿。

**原因：** 高频输出需要批量写入和节流优化。

**建议：**
- 限制同时打开的终端数量
- 适当减小滚动缓冲行数
- 避免在后台标签中运行高输出命令

## 配置文件损坏

**限制：** 如果配置文件（config.json、projects.json、layout-state.json）损坏，应用会使用默认配置。

**容错机制：**
- JSON 解析失败时使用默认值
- 缺失字段时使用默认值填充
- 无效值时使用默认值

**恢复方法：** 删除损坏的配置文件，应用会重新创建默认配置。

## 下一步计划

以下功能计划在后续版本中实现：

- v1.0：命令面板、设置页增强、主题定制、右键菜单
- v1.5：布局模板、拖拽重排、浮动面板、Shell 集成增强
- v2.0：插件系统、SSH 管理、命令块 UI
